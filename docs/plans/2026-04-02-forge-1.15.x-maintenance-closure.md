## Closure Decision

Forge entered maintenance mode after the `1.15.0` stable release.
That closure remains a historical milestone, but it is no longer the current roadmap line.

Current policy now lives in:

- `docs/plans/2026-04-11-forge-slim-refactor-v2.md` for the active surface-slim roadmap tranche
- `docs/current/architecture.md`, `docs/current/operator-surface.md`, and `docs/current/install-and-activation.md` for current maintainer guidance
- `packages/forge-core/references/target-state.md` for the shipped `1.15.x` closure target plus the explicit reopen line
- `packages/forge-core/references/reference-map.md` for the current reading order
- `README.md`, `docs/release/public-readiness.md`, `docs/release/release-process.md`, and `CHANGELOG.md` for the public stable-release story
- this document for the original maintenance-closure evidence chain

All roadmap files under `docs/plans/` should now be one of:

- `historical`
- `historical maintenance closure`
- `current roadmap`

## Allowed Maintenance Work

Treat these as maintenance, not as a new roadmap line:

- release-facing doc sync, version sync, and changelog hygiene
- compatibility repairs for host adapters, generated artifacts, or stable workflow-state contracts
- verification hardening when a real drift path or regression gap is found
- narrow operator clarity improvements that reduce ambiguity without adding a new product surface

## Reopen Signals

Reopen the roadmap only if:

- runtime health, packet execution, or host-neutral dispatch contracts fail in real repos and bounded maintenance work cannot contain the problem
- release-facing drift survives the current verification stack, proving the closure checks are materially incomplete
- a proposed addition changes the behavior bar enough that calling it maintenance would hide a real product decision

## Superseded Planning Inputs

The following now remain as historical reference only:

- the process-first roadmap and implementation-plan documents from `2026-03-29`
- the solo-profile checklist and `1.11.x`, `1.12.x`, `1.13.x`, and `1.14.x` release-line roadmaps
- the `1.14.x` engineering scorecard and the build-throughput candidate plan
- older idea backlogs and proposals that were useful during active roadmap shaping but are not current work queues anymore

When a future change needs more than maintenance:

1. update `packages/forge-core/references/target-state.md` first
2. create a new explicit roadmap document instead of silently reactivating an old one
3. tighten verification before changing the public product story
