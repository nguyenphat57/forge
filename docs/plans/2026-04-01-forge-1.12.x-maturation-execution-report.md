# Forge 1.12.x Maturation Execution Report

Date: 2026-04-01
Status: implemented
Source plan:
- `docs/plans/2026-04-01-forge-1.12.x-maturation-roadmap.md`
- `docs/plans/2026-04-01-forge-1.12.x-maturation-implementation-plan.md`

## Outcome

All planned `1.12.x` maturation phases were implemented in the current worktree:

- Phase 1: workflow-state-backed release behavior is now paired with an explicit canonical release-tier story and a bounded adoption loop
- Phase 2: operator and architecture surfaces now align on the same release-tail semantics
- Phase 3: `help` and `next` now read release-tail state directly, and the operating-model references explain the intended `1.12.x` posture
- Phase 4: dashboard visibility now surfaces release tier plus adoption signal, and the deferred `1.13+` boundary is explicit in tracked docs

## Delivered Changes

### Roadmap Closure Hardening

- `docs/plans/2026-04-01-forge-1.12.x-maturation-roadmap.md`
  - freezes `1.12.0` to one canonical four-tier release vocabulary
  - distinguishes the `1.12.0` cut sweep from the `1.12.x roadmap-complete` sweep
  - upgrades Commit 7, Commit 9, and Commit 10 proof from loose token grep to structured content checks
- `docs/plans/2026-04-01-forge-1.12.x-maturation-implementation-plan.md`
  - mirrors the stronger closure rules so the execution packet does not drift from the roadmap
- `packages/forge-core/references/target-state.md`
  - turns the deferred `1.13+` boundary into a rationale-carrying list instead of a plain backlog sketch

### Phase 1: Behavior Backbone

- `packages/forge-core/scripts/release_profile_contract.py`
  - freezes the canonical `1.12.x` release-tier policy
  - adds compatibility, migration, rollback, and follow-up packet metadata for the current four canonical tiers
- `packages/forge-core/scripts/release_readiness.py`
  - emits `release_story`, `migration_guidance`, `rollback_guidance`, and `follow_up_packet`
  - turns live adoption state into `planned`, `pending-adoption`, `supported`, `monitoring`, or `needs-attention`
- `packages/forge-core/scripts/record_adoption_check.py`
  - bounds adoption feedback to explicit friction categories and release actions
- `packages/forge-core/scripts/workflow_state_support.py`
  - persists adoption friction categories and release actions into workflow-state

### Phase 2: Operator And Architecture Alignment

- `packages/forge-core/workflows/execution/adoption-check.md`
  - defines bounded signal shapes and direct Forge actions
- `packages/forge-core/workflows/execution/release-readiness.md`

- `docs/architecture/adapter-boundary.md`
  - forbids host adapters from inventing new release-tier semantics
- `docs/architecture/monorepo.md`
  - makes `release-readiness.md` and `adoption-check.md` canonical in `forge-core`
- `packages/forge-core/SKILL.md`
- `packages/forge-codex/overlay/SKILL.md`
- `packages/forge-antigravity/overlay/SKILL.md`

### Phase 3: State-First Guidance
