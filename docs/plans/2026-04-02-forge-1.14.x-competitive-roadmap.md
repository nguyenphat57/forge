## Role Of This Document

- translate competitive pressure into a bounded `1.14.x` implementation program
- complement the `1.14.x` engineering scorecard instead of replacing it
- keep Forge's product thesis intact while deciding what to absorb, what to defer, and what to reject

This roadmap is not:

- a public benchmark claim
- a repo-to-repo feature checklist
- permission to copy another system's whole identity into Forge

## Goal

Make Forge more competitive as a solo-dev execution system by increasing execution throughput, reducing avoidable operator friction, and strengthening runtime leverage without weakening proof, brownfield safety, or host neutrality.

## Product Theme

`1.14.x` should be about:

`execution throughput without contract drift`

That means:

- lighter handling for truly small work
- richer packet execution for medium and large work
- stronger runtime and browser reliability where those tools materially accelerate proof
- better host portability through stronger contracts, not thicker wrappers
- more useful continuity and context loading without turning Forge into an everything-platform

## North-Star Constraint

Use the `target-state.md` guardrails throughout:

- stay process-first
- stay brownfield-safe
- stay artifact-driven
- stay evidence-backed
- stay useful for a solo developer in a real repo

If a `1.14.x` idea improves throughput by weakening those constraints, reject it.

## Competitive Pressures That Matter

### gstack

Pressure:
- stronger real-browser and release-cockpit ergonomics
- higher perceived ship velocity for Claude-first operators
- persistent browser runtime treated as a first-class execution surface

What Forge should absorb:
- runtime health and reliability for browser-backed flows
- clearer merge, review, and release handoff ergonomics
- more usable execution cockpit behavior without making the dashboard the source of truth

What Forge should not copy:
- Claude-first identity as the product center
- "virtual engineering team" theater as the primary mental model
- host-specific browser assumptions inside core contracts

### get-shit-done

Pressure:
- lower-friction small-task and solo-builder workflow
- clearer assumptions-first planning
- strong value density without enterprise ceremony

What Forge should absorb:
- explicit fast-lane rules for low-risk work
- assumptions-driven context capture for predictable repos
- verification-aware planning that feels lighter, not weaker

What Forge should not copy:
- convenience posture that normalizes broad permission skipping as the default answer
- "frictionless at any cost" if it erodes brownfield safety

### CodyMaster

Pressure:
- deeper continuity and memory ambition
- broader lifecycle coverage across docs, design, and long-running knowledge surfaces
- stronger perception of a connected AI operating system

What Forge should absorb:
- bounded continuity spine improvements
- progressive context loading where it reduces token waste and routing drift
- stronger linkage between packet state, learnings, and codebase summaries

What Forge should not copy:
- feature sprawl into content, growth, enterprise, and generalized AI ecosystem surfaces
- self-healing or cloud-brain complexity without a narrow Forge payoff
- breadth-first platform identity that weakens Forge's kernel clarity

### Spec Kit

Pressure:
- spec and plan discipline with a clean extension model
- strong "workflow as framework" story

What Forge should absorb:
- cleaner extension or preset surface for bounded workflow customization
- stronger artifact contracts for packet templates and implementation archetypes

What Forge should not copy:
- spec bureaucracy for work that does not need it
- extension surface as a reason to weaken the default core delivery path

### Superpowers

Pressure:
- strong workflow activation discipline
- clear subagent and TDD posture
- host reach through a lightweight skill surface

What Forge should absorb:
- more explicit workflow activation rules for execution slices
- clearer review and TDD expectations where a harness exists

What Forge should not copy:
- skill choreography without durable state
- dependence on chat-time skill sequencing as a substitute for packet and workflow-state artifacts

### Antigravity Kit

Pressure:
- zero-learning-curve onboarding
- easy specialist routing and fast first-use payoff

What Forge should absorb:
- simpler "start here" guidance for new solo operators
- friendlier first-run guidance for low-complexity workflows

What Forge should not copy:
- shallow template-pack behavior that lacks strong proof and state contracts

## Non-Goals

- do not add a new release-tier taxonomy
- do not build a content factory, growth engine, or enterprise integration layer
- do not turn the dashboard into a telemetry-heavy command center
- do not make browser QA mandatory for non-browser work
- do not turn Forge into a host-specific product for Claude, Codex, or Gemini alone
- do not replace workflow-state with a parallel memory system
- do not build cloud-synced "soul" or self-healing infrastructure unless a narrow, local, execution-facing use case proves necessary first

## Program Shape

`1.14.0` should ship the minimum set that makes the competitive response real.

Recommended interpretation:

- `1.14.0` = fast lane + packet graph + runtime health + host capability contract + target-state sync
- `1.14.x roadmap-complete` = plus bounded continuity spine, bounded extension surface, and thin onboarding improvements

## Sequencing Rules

- reduce small-task friction before adding more packet complexity to light work
- strengthen packet graph and merge protocol before widening more parallelism
- improve browser-runtime trust before widening browser-backed proof usage
- freeze host capability semantics before broadening adapter-specific convenience
- add continuity depth only after the execution kernel stays clearly smaller than a platform-style memory stack
- if a competitive response mainly increases breadth instead of leverage, defer it

## Release Bar

`1.14.0` should only ship once:

- small low-risk work has a clearer fast path that preserves proof-before-claims
- medium and large work can express packet dependency, merge intent, and overlap risk more explicitly than `1.13.0`
- browser-capable runtime paths have explicit health or doctor-style diagnostics
- host capability routing is easier to extend without changing packet semantics
- the canonical `1.14.x` target and deferred boundary are promoted in tracked docs instead of staying implied
- the public and operator story remains process-first rather than "AI team in a box"

## Entry Gate For 1.14.x

Do not start Commit 1 until:

- the `1.13.0` cut remains green on the canonical repo gate
- there is no accepted open regression in packet summaries, `help/next` state resumption, or runtime-tool flag forwarding
- the `1.13.x` packet contract from `track_execution_progress.py` is treated as the frozen migration baseline for `1.14.x`
- the specific deferred-boundary items being promoted from `target-state.md` are explicitly named before implementation starts
- the first implementation packet names one owner each for fast-lane contract, packet-graph schema, runtime diagnostics, and target-state sync

If any gate fails, record the blocker and pause `1.14.x` instead of coding around it.

## Fast Lane Contract V1

The competitive answer to GSD-style low-friction work must not become a second execution system.

Source of truth:

- fast-lane work still persists through `track_execution_progress.py`
- the packet remains the authoritative record even when the field set is smaller than a full medium or large packet
- `workflow_state_summary.py`, `help_next_support.py`, and dashboard surfaces stay read models only

Eligibility:

- low blast radius
- narrow owned scope
- no packet graph or multi-lane dispatch needed
- no schema migration, auth boundary, payment boundary, public interface break, or release-critical tail work

Required packet fields for `packet_mode = fast-lane`:

- `packet_id`
- `packet_mode`
- `label`
- `goal`
- `status`
- `exact_files_or_paths_in_scope`
- `owned_files_or_write_scope`
- `baseline_or_clean_start_proof`
- `proof_before_progress`
- `verification_to_rerun`
- `residual_risk`
- `next_steps`

Rules:

- fast lane may reduce artifact overhead, but it may not skip clean-start proof, verification rerun, or residual-risk capture
- `help` and `next` must read the same packet/state surfaces they already use for packetized execution; no fast-lane-only state file may be introduced
- the route into and out of fast lane must be explicit enough that a stalled slice can be escalated into full packet mode without lossy migration

## Packet Graph Schema V1

The packet-graph response to gstack and CodyMaster pressure must keep one canonical schema.

Source of truth:

- `track_execution_progress.py` owns the authoritative packet record
- `track_chain_status.py` is the controller read model for chain and merge state
- `workflow_state_summary.py`, `help_next_support.py`, and `resolve_help_next.py` summarize graph state but do not invent graph fields
- `route_delegation.py` emits graph-aware packet templates derived from the canonical schema instead of defining a parallel one

Canonical graph fields to add or freeze for `1.14.0`:

- `packet_mode`
- `depends_on_packets`
- `unblocks_packets`
- `merge_target`
- `merge_strategy`
- `overlap_risk_status`
- `write_scope_conflicts`
- `merge_readiness`
- `review_readiness`
- `completion_gate`

Compatibility rule:

- `parent_packet` may remain as a compatibility alias where needed
- after Commit 2, new producers and consumers should prefer graph fields over inferring dependency from chat or from a parent-only model
- overlap and merge summaries must derive from these graph fields instead of introducing a third naming layer in `track_chain_status.py` or `workflow_state_summary.py`

## Runtime Health Scope For 1.14.0

Commit 3 must stay bounded:

- in scope: repo-local and installed-runtime diagnostics for browser-capable toolchains that Forge already invokes
- in scope: failure taxonomy, stale-state detection, and proof-link clarity between runtime logs and packet summaries
- out of scope for `1.14.0`: rollout ledgers, broad canary expansion, or host-wide observability systems beyond what the runtime invocation path needs

## Ownership And Verification Rule

The entry gate owner list must become commit-local accountability, not just a kickoff note.

- every implementation packet must restate one primary owner for the commit contract and one verification owner for the release bar
- in solo-dev mode the same person may hold both roles, but the packet must still name both responsibilities explicitly
- if a companion skill, subagent, or host-specific wrapper contributes changes, the commit owner remains responsible for final contract coherence and revert decisions
- no commit that changes runtime behavior may close with docs-only evidence; each such commit must add or update at least one regression or acceptance-level proof surface

## Rollback And Revert Protocol

If a must-have commit regresses `help/next`, packet summaries, runtime-tool invocation, or packet-state readability:

- revert to the last green cut line instead of layering more compatibility prose on top of a broken behavior change
- keep new packet-graph fields additive until the `1.14.0` cut so graph-aware producers or readers can be disabled without corrupting the `1.13.x` packet baseline
- prefer falling back to the last green `1.13.x`-compatible summary path over inventing partial graph semantics in `workflow_state_summary.py` or `help_next_support.py`
- record the failing scenario, the revert point, and the re-entry condition in the active implementation packet before restarting work

## Definition Of Done

`1.14.x` is successful only when:

- Forge feels lighter than `1.13.0` on truly small work without being looser
- Forge feels stronger than `1.13.0` on packet decomposition, merge flow, and overlap control
- browser-backed proof is operationally healthier, not merely more available
- host portability gets stronger while core contracts stay singular
- continuity gets more useful without turning Forge into a breadth-first platform
- the promoted `1.14.x` target and deferred boundary are visible in tracked references instead of staying roadmap-only
- the competitive response makes Forge more itself, not less
