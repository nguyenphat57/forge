## Roadmap Role

- this document is the committed `1.12.x` operating-maturity roadmap
- `docs/plans/2026-04-01-forge-1.12.x-ideas.md` remains input and backlog only; an idea does not become release-bar work until it is promoted here

## Goal

Move Forge from a hardened release-credible system into a mature solo-dev operating model that is easier to operate, easier to explain, and more consistent from routing through post-release follow-up.

## Non-Goals

- do not reopen `1.11.x` security or provenance fixes unless regressions are found
- do not add broad new companion breadth
- do not turn Forge into an enterprise SRE platform
- do not introduce a second competing operating model next to the solo-profile contract

## Release Bar

`1.12.0` should only ship once:

- workflow-state is the default backbone for medium+ and release-sensitive work
- release tiering is more explicit than the current broad `solo-public` posture
- adoption-check is a normal part of the post-release loop instead of a rarely used tail stage
- operator surfaces across adapters tell one consistent story about routing, gates, and aliases
- architecture docs make the core, overlay, generated-artifact, and runtime-state layers easy to understand

## Strategy Shift

`1.11.x` was about hardening trust.  
`1.12.x` is about operational maturity.

That means:

- less hidden state
- less manual operator choreography
- less ambiguity in release posture
- better explanation of how Forge actually works

## Sequencing Rules

- Make workflow-state default before layering richer release tiers on top of it.
- Do not add more release policy nuance until the state backbone can carry it cleanly.
- Operator-surface cleanup must follow the actual routing and state contract, not invent parallel UX.
- If a feature changes how readiness is decided, update the docs, scripts, and adapter surfaces in the same release train.
- Keep `1.12.x` centered on solo-dev operating maturity, not feature sprawl.

## Dependencies And Risks

- `1.11.x` hardening remains a hard dependency; if a hardening regression reopens during `1.12.x`, pause maturity work, land the regression fix first, then restack the affected `1.12.x` slice
- Commit 2 depends on Commit 1 making workflow-state trustworthy enough to carry release posture without chat reconstruction
- Commit 4 depends on Commit 2 freezing the canonical release-tier vocabulary or compatibility story first; adapter cleanup should reflect that contract, not invent it
- Commit 6 depends on the workflow-state and release-tier artifacts from Commits 1 to 3 being stable enough for `help` and `next` to consume directly
- if a commit starts pulling in new host-specific behavior instead of aligning shared contracts, cut or defer that scope to keep `1.12.x` focused

- freeze the canonical `1.12.0` tier set at `internal-shared`, `internal-critical`, `public-controlled`, and `public-broad`

### Commit 3: Make Adoption Check Actionable

Deliver:
- keep adoption-check richer than a thin note recorder without turning it into a mini analytics system
- define a bounded signal contract for solo-dev use: signal impact, confidence, concrete friction, and next action
- keep friction categories concrete enough to drive action, for example `regression`, `confusion`, `adoption lag`, or `environment issue`, instead of open-ended commentary
- feed adoption-check back into readiness and release follow-up instead of leaving it as a detached artifact
- produce a structured follow-up signal that `help` or `next` can consume later instead of wiring operator guidance in parallel here
- define escalation only as release-tail action escalation inside Forge, such as `monitor`, `follow-up fix`, `rollback`, or `re-run readiness`, not escalation to another team or org layer

## Definition Of Done

`1.12.x` is done only when:

- the `1.12.0` cut sweep is green for Commits 1 to 5, and the roadmap-complete sweep is green before claiming Commits 6 to 10 are also finished
- workflow-state feels like the default operating backbone, not an advanced option
- release posture is explicit enough that solo-dev rollout decisions are not forced into one broad public bucket

- the architecture is explainable from tracked docs and shipped artifacts without relying on maintainers to narrate it live
