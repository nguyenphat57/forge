## Roadmap Role

- this document is the committed `1.13.x` roadmap for build-process enhancement
- prioritize actual build throughput and resumption quality over new release-tail or runtime-trust surfaces
- treat host-native subagents and browser tooling as accelerators that remain subordinate to Forge's proof and ownership contract

## Goal

Make Forge materially stronger at medium and large build execution without changing the product thesis: process-first, brownfield-safe, evidence-backed orchestration.

## Non-Goals

- do not redesign release tiers or release-tail workflow again
- do not make native subagents mandatory in `forge-core`
- do not make browser QA a default requirement for non-browser work
- do not introduce a dashboard-first experience before packet state is already useful
- do not mix installed-runtime doctor, rollout ledger, or publish-packet work into the first build-process tranche unless a regression forces it

## Release Bar

`1.13.0` should only ship once:

- medium and large build work can be represented as explicit packets with scope, ownership, proof, blockers, and reopen rules
- packet and chain state are strong enough that `help` and `next` can resume the real build loop from artifacts instead of chat memory
- host capability differences only change dispatch mode and fallback, not the core packet contract
- browser-backed verification can be classified and invoked for qualifying packets without weakening the core proof model
- public build-throughput messaging is supported by verified behavior, not aspirational docs

## Strategy Shift

`1.11.x` hardened trust.
`1.12.x` matured the operating model.
`1.13.x` should make build execution feel as intentional as release execution.

That means:

- explicit build packets instead of loose checkpoints
- controller-visible chain state instead of scattered artifacts
- safe host-specific acceleration on top of one core contract
- targeted browser verification when it shortens proof loops
- operator guidance that reads the active build state first

## Boundary With Other 1.13+ Work

Keep the following out of the first build-process line unless a later slice explicitly promotes them:

- installed-runtime doctor expansion
- host rollout ledgers
- runtime canary growth beyond the current release contract
- generated release publish packets
- broader companion breadth as a substitute for fixing the core build loop

## Canonical Packet Schema V1

For `1.13.0`, the packet contract needs one clear source of truth before dispatch or browser work widens:

- the authoritative per-packet record is the persisted execution packet JSON emitted by `track_execution_progress.py`
- `route_delegation.py` owns dispatch packet templates and lane semantics, but it must derive from the canonical packet schema instead of inventing a second schema
- `track_chain_status.py` is controller-level chain state, not the packet source of truth
- `workflow_state_summary.py`, `help_next_support.py`, and `resolve_help_next.py` are read models only; they may summarize packet state, but they must not fabricate packet fields

The canonical packet field set for `1.13.0` should converge on:

- `packet_id`
- `parent_packet`
- `project`
- `label`
- `goal`
- `current_stage`
- `status`
- `completion_state`
- `source_of_truth`
- `exact_files_or_paths_in_scope`
- `owned_files_or_write_scope`
- `baseline_or_clean_start_proof`
- `red_proof`
- `proof_before_progress`
- `verification_to_rerun`
- `browser_qa_classification`
- `browser_qa_scope`
- `browser_qa_status`
- `blockers`
- `residual_risk`
- `out_of_scope_for_this_slice`
- `reopen_conditions`
- `next_steps`

Compatibility rule:

- Commit 1 may preserve temporary aliases for existing keys such as `task`, `stage`, `scope_paths`, `baseline_proof`, `risks`, and `next`
- after Commit 1, new producers and consumers should converge on the canonical names above instead of adding a third naming layer
- the producer or consumer mismatch list must explicitly cover at least `task` vs `label`, `stage` vs `current_stage`, `next` vs `next_steps`, and `risk` vs `residual_risk`

## Sequencing Rules

- Freeze packet state before widening dispatch.
- Freeze dispatch contract before teaching adapters richer acceleration behavior.
- Add browser-QA routing only after packets can represent why it is needed and where its proof lands.
- Treat browser-proof binding as a follow-on slice, not a `1.13.0` ship blocker, until packet eligibility and state consumption are stable.
- Make `help` and `next` consume build-state artifacts after those artifacts are stable enough to trust.
- Refresh public positioning only after the underlying capability is verified.
- Keep an optional dashboard out of the critical path until the non-dashboard surfaces prove insufficient.

## Dependencies And Risks

- `1.12.x` workflow-state work is the hard prerequisite; if its contract reopens, pause `1.13.x` build work and land the regression fix first.
- Commit 2 depends on Commit 1 making packet state credible enough to summarize.
- Commit 4 depends on Commit 3 freezing host-neutral dispatch semantics before adapter wrappers try to explain them.
- Commit 5 depends on Commits 1 to 4 creating a packet and chain model that can classify browser-QA need without parallel sidecars.
- Commit 8 depends on Commit 5 freezing browser-QA classification and packet markers before proof binding widens runtime-tool integration.
- If a slice starts adding host-specific behavior to core instead of strengthening shared contracts, cut or defer that scope.
- If a slice improves visibility but cannot show a merge, resume, or verification benefit, it is likely telemetry drift and should be challenged.

## Effort Guide

Use this only for solo-dev planning and cut decisions, not as a promise:

| Commit | Effort | Why |
|--------|--------|-----|
| 1 | `M` | packet contract touches execution artifacts, delegation surfaces, and round-trip tests |
| 2 | `M` | chain-state and summary changes must line up with `help` and `next` consumption |
| 3 | `M` | host-neutral dispatch changes routing semantics and host capability mapping |
| 4 | `S` | workflow and adapter wording must follow the frozen contract instead of inventing local behavior |
| 5 | `S` | browser-QA eligibility stays bounded if it only classifies packets and sets packet markers |
| 6 | `S` | state-first operator guidance builds on stabilized artifacts instead of new engine work |
| 7 | `S` | public positioning and release-facing wording should only trail verified behavior |
| 8 | `M` | browser-proof binding crosses runtime-tool invocation, packet summaries, and operator consumption |
| 9 | `S` | canonical reference sync is bounded, but it needs stable terminology from the earlier commits |
| 10 | `S` | dashboard work stays optional and should remain a thin view over existing packet artifacts |

## Definition Of Done

`1.13.x` is done only when:

- the `1.13.0` cut sweep is green before claiming the first public build-process enhancement release
- build packets are the default control surface for medium and large implementation work, not an optional side channel
- host-native acceleration strengthens throughput without weakening ownership, proof, or brownfield safety
- when browser-backed verification is used, it is visible in the same artifact story as other proof instead of drifting into isolated runtime logs
- operator guidance can resume real build chains from persisted state
- public and operator-facing docs describe the stronger build loop without overstating what Forge can actually prove
