## Goal

Close the highest-value gaps that remain after `1.10.2`, especially where Forge still looks stronger in demos than in hard release or routing reality.

## Non-Goals

- do not introduce a new product thesis
- do not expand companion breadth
- do not redesign the release model while core hardening gaps are still open
- do not mix ergonomics polish with security or provenance fixes in the same commit unless the contract truly depends on it

## Release Bar

`1.11.0` should only ship once:

- secret scanning covers tracked and untracked release-relevant files
- release artifacts can distinguish clean-tree and dirty-tree provenance
- solo-profile and stage-chain behavior is visible in adapter-facing docs instead of being hidden only in scripts
- adapter runtime bundle contracts are explicitly tested at the materialized `dist/` layer

## Sequencing Rules

- Fix security and provenance gaps before polishing docs or aliases.
- Keep each commit independently reviewable and testable.
- When a change affects bundle materialization, assert the contract at `dist/` level instead of only checking overlay source files.
- Docs-only commits should follow behavior commits that they describe; do not front-load documentation for behavior that still does not exist.
- If an ergonomic alias is intentionally rejected, encode that decision in docs and tests instead of leaving it ambiguous.

## Definition Of Done

`1.11.x` hardening is only done when:

- security and provenance gaps are actually closed, not just documented
- solo-profile behavior is discoverable from adapter docs and shipped bundle contracts
- release-tail workflows read as intentional parts of the system, not hidden sidecars
- adapter/runtime contract disputes can be resolved from tests and dist artifacts instead of opinion
