## Theme

`1.12.x` should make the installed runtime as trustworthy as the repo and `dist/` layers.

`1.11.0` hardened release contracts, stage-state continuity, and materialized bundle verification.
The next gap is no longer "can Forge build the right bundle?" but "can an operator trust the live host after install without manual forensic checks?"

## What `1.12.x` Should Optimize For

- close the gap between `dist/` truth and installed-runtime truth
- make host activation and runtime-state health observable, not implicit
- reduce manual rollback or inspection work after install
- keep the core-vs-adapter boundary intact while improving real rollout confidence

## Non-Goals

- do not introduce a new companion lane
- do not redesign core routing again right after `1.11.0`
- do not expand host-specific UX in `forge-core` when the engine can stay shared
- do not mix runtime trust work with broad product-thesis changes

## Candidate Ideas

### Idea 1: Make Installed-Runtime Health A First-Class Doctor Surface

Why:
- today the canonical release gate stops at repo verify, build, and install
- the operator still has to manually inspect `INSTALL-MANIFEST.json`, global host files, and resolver behavior to know whether the live host is actually healthy
- the real `1.11.0` rollout exposed that this is the current trust gap

Deliver:
- add an installed-runtime check mode that validates:
  - installed version and `git_revision`
  - bundle fingerprint vs installed target
  - host activation result for `AGENTS.md` or `GEMINI.md`
  - state-root existence and expected layout
  - installed `resolve_preferences.py` execution
- let `doctor` distinguish bundle health, host activation health, and state health

Primary files:
- `packages/forge-core/workflows/operator/doctor.md`
- `packages/forge-core/scripts/doctor.py`
- `packages/forge-core/scripts/doctor_runtime.py`
- `scripts/install_bundle_runtime.py`
- `tests/test_install_bundle_antigravity_host.py`
- `tests/test_install_bundle_codex_host.py`

Evidence:
- `docs/release/release-process.md` still treats post-install smoke as optional
- `scripts/install_bundle_runtime.py` writes `INSTALL-MANIFEST.json` but there is no canonical post-install verifier
- the actual `1.11.0` rollout needed manual checks against installed manifests and global host entry files

### Idea 2: Add Safe Repair For Corrupt Runtime Preference State

Why:
- a runtime can be installed correctly while still behaving poorly because host state is corrupt
- that ambiguity weakens operator trust because the bundle looks broken even when the failure is in persisted state
- the recent Codex runtime check returned `WARN` because `extra_preferences.json` was invalid JSON

Deliver:
- add a safe repair path for malformed `preferences.json` or `extra_preferences.json`
- back up corrupt state before repair
- separate "bundle install is healthy" from "persisted host state is unhealthy"
- optionally let `doctor` emit a repair command or machine-readable remediation packet

Primary files:
- `packages/forge-core/scripts/preferences.py`
- `packages/forge-core/scripts/write_preferences.py`
- `packages/forge-core/scripts/doctor.py`
- `packages/forge-core/tests/preferences_test_loading.py`
- `packages/forge-core/tests/test_doctor.py`

Evidence:
- installed Codex resolver returned `WARN` because global `extra_preferences.json` was invalid
- preference migration tests exist, but runtime-state corruption is not yet surfaced as its own first-class remediation story

### Idea 3: Promote Gemini Host Activation To The Same Release Surface As Codex

Why:
- the implementation and tests already support `--activate-gemini`
- the public install story still explains Codex host takeover more clearly than Gemini host takeover
- that mismatch makes the Gemini rollout look less first-class than it actually is

Deliver:
- update install and release docs so Gemini host activation is explicit, not implied
- keep standard install snippets aligned with the real activation contract
- make it obvious when a rollout should install only the bundle versus install plus host takeover

Primary files:
- `docs/release/install.md`
- `docs/release/release-process.md`
- `tests/test_install_bundle_antigravity_host.py`
- `scripts/install_bundle.py`
- `scripts/install_bundle_host.py`

Evidence:
- `docs/release/install.md` standard flow shows `forge-antigravity --build` but not `--activate-gemini`
- `tests/test_install_bundle_antigravity_host.py` already proves `--activate-gemini` is real and supported

### Idea 4: Persist A Host Rollout Ledger Instead Of Leaving Install History In Scattered Files

Why:
- install backups and per-target manifests exist, but there is no single history surface for "what was installed, where, when, from which build, and with which activation result"
- once multiple upgrades happen, it becomes harder to answer rollback and adoption questions from one place

Deliver:
- persist a small rollout ledger under adapter-global state or a repo-side release artifact
- record:
  - bundle
  - target
  - version
  - `git_revision`
  - fingerprint
  - activation flags
  - backup path
  - install timestamp
- keep the ledger usable by `doctor`, release-readiness, and rollback guidance

Primary files:
- `scripts/install_bundle_runtime.py`
- `scripts/install_bundle_report.py`
- `packages/forge-core/scripts/doctor.py`
- `packages/forge-core/scripts/resolve_rollback.py`
- related install tests

Evidence:
- `scripts/install_bundle_runtime.py` writes only the current `INSTALL-MANIFEST.json`
- `.install-backups/` preserves old copies, but not a queryable rollout history

### Idea 5: Refine Doctor's Runtime Warnings So Optional Tools Do Not Look More Broken Than They Are

Why:
- Forge already knows runtime tools are optional
- the current doctor behavior can still produce warnings when bundle-neighbor resolution works but no registry path exists
- this was explicitly left as acceptable residual risk in the Phase 1 report, which makes it a strong follow-up candidate

Deliver:
- downgrade or remove the false-warning case when runtime tools resolve successfully without an explicit registry
- separate "optional and unresolved" from "misconfigured and expected to exist"
- keep strict mode meaningful instead of noisy

Primary files:
- `packages/forge-core/scripts/doctor_runtime.py`
- `packages/forge-core/scripts/runtime_tool_support.py`
- `packages/forge-core/tests/test_doctor.py`
- `docs/PROJECT_PHASE1_REPORT_2026-03-28.md`

Evidence:
- `docs/PROJECT_PHASE1_REPORT_2026-03-28.md` calls out this warning as an explicit residual risk
- `packages/forge-core/scripts/doctor_runtime.py` marks missing registry resolution as `WARN` even though runtime tools are optional

### Idea 6: Add An Installed-Runtime Canary Path

Why:
- `verify_repo.py` and bundle verify prove source and `dist/`
- install tests in temp directories prove the install logic
- neither one proves that the operator's real `.codex` or `.gemini` host takeover remains healthy after a live upgrade

Deliver:
- add a lightweight installed-runtime canary that runs against the live installed bundle path
- start with low-risk checks such as:
  - resolver execution
  - host entrypoint rendering sanity
  - expected state root resolution
  - manifest/version consistency
- keep it opt-in so release policy does not require host-specific soak for every release

Primary files:
- `docs/release/release-process.md`
- `scripts/install_bundle.py`
- `tests/test_install_bundle_antigravity_host.py`
- `tests/test_install_bundle_codex_host.py`
- possibly new host-canary helpers under `packages/forge-core/scripts/`

Evidence:
- current release process explicitly keeps extra runtime evidence optional
- recent live rollout still required manual post-install checks outside the canonical verify pipeline

### Idea 7: Generate A Small Release-Publish Packet From Existing Manifests

Why:
- Forge already has strong manifests, fingerprints, transitions, and install metadata
- release publication still depends on stitching together notes, install commands, and bundle facts by hand
- a thin publish packet would improve operator trust without changing runtime behavior

Deliver:
- generate a small release packet from `VERSION`, `CHANGELOG.md`, `BUILD-MANIFEST.json`, and package metadata
- include:
  - version
  - commit
  - bundle list
  - install snippets
  - notable activation flags
  - release-facing verification summary
- keep it deterministic and release-facing, not a marketing layer

Primary files:
- `scripts/build_release.py`
- `docs/release/release-process.md`
- `docs/release/package-matrix.json`
- release contract tests

Evidence:
- release metadata is present across several sources, but there is no single generated publish packet
- past release bodies and notes are still maintained as hand-written documents

## Recommended Shape

If `1.12.0` should stay focused, the strongest cut is:

- Idea 1: installed-runtime doctor
- Idea 2: preference-state repair
- Idea 3: Gemini host activation parity
- Idea 5: doctor runtime-warning refinement

That would make `1.12.0` feel like one coherent release:
"Forge can now prove and repair the live runtime, not just the repo."

## Stretch Or Follow-Up Cut

If the release wants a broader operational story, add:

- Idea 4: rollout ledger
- Idea 6: installed-runtime canary
- Idea 7: generated release-publish packet

These are valuable, but they are easier to cut than the core runtime-trust slice above.

## Exit Signal For `1.12.x`

`1.12.x` is successful if an operator can:

- install or upgrade a host bundle
- run one canonical runtime-health check
- distinguish bundle issues from state corruption
- trust the host activation result without manual file archaeology
- explain rollback or follow-up from persisted rollout evidence instead of memory
