## Role Of This Document

- provide an engineering-facing scorecard for the current Forge axes instead of a release-note recap
- compare `1.12.0` and `1.13.0` on one weighted frame
- identify the highest-leverage `1.14.x` investments without reopening settled `1.12.x` and `1.13.x` contracts casually

## Scoring Method

- score each axis on a `10` point scale where `10` means "strong, coherent, and low-drift for Forge's current thesis", not "perfect forever"
- weight each axis by engineering value to Forge's product direction, not by how visible it is in demos
- treat the scorecard as a planning instrument, not benchmark telemetry
- keep the north-star thesis from `target-state.md`: process-first, brownfield-safe, artifact-driven, evidence-backed execution

Weighted score formula:

```text
weighted contribution = axis weight * axis score / 10
weighted total = sum of all weighted contributions
```

## Weighted Totals

- `1.12.0`: `82.3 / 100`
- `1.13.0`: `91.3 / 100`
- Net gain: `+9.0`

Interpretation:

- `1.12.0` made Forge operationally mature and release-credible
- `1.13.0` moved build execution much closer to the same level of intentionality and resumability
- the remaining gap is now less about release discipline and more about execution throughput, host/runtime leverage, and keeping the system light on low-risk work

## Axis Reviews And 1.14.x Recommendations

### 1. Small-Task Ergonomics

Current read:
- Forge still handles small work reasonably well
- the system is now optimized more clearly for medium and large execution than for trivial edits
- the main risk is process spillover, not current failure

Why the score moved:
- `1.13.0` added stronger packet machinery for the right reasons
- that machinery does not yet come with an equally explicit "stay light" contract for low-risk work

`1.14.x` recommendation:
- define a clear fast-lane for tasks with low blast radius and low ambiguity
- make packet creation conditional on explicit risk, not on generic build intent alone
- keep the proof bar intact while reducing artifact overhead for obviously bounded changes

Success signal:
- small tasks stay evidence-backed without feeling like miniature large-task ceremonies

### 2. Medium Or Large Build Execution

Current read:
- this is the biggest `1.13.0` improvement
- build packets, chain state, merge points, and proof reruns make execution materially more intentional
- Forge now has a credible build-side operating model, not just a release-side one

Why the score moved:
- `track_execution_progress.py` became the packet source of truth
- summaries, guidance, and dispatch now read that packet model instead of working around missing structure

`1.14.x` recommendation:
- add explicit packet dependency graph semantics, not just parent-child hints
- standardize merge-back protocol and packet completion criteria by slice type
- define reusable packet templates for common execution archetypes such as bugfix, refactor, feature slice, and risky migration

Success signal:
- medium and large builds can be decomposed, resumed, merged, and reviewed with less operator improvisation

### 3. Workflow-State Backbone

Current read:
- workflow-state is now one of Forge's strongest foundations
- the current gap is less about adoption and more about long-term schema durability

Why the score moved:
- `1.12.0` made workflow-state the default backbone
- `1.13.0` increased the amount of behavior that depends on it, especially packet summaries and browser evidence

`1.14.x` recommendation:
- add explicit schema versioning and migration rules for packet-bearing workflow-state
- improve stale-state detection and legacy-summary repair
- expand regression coverage for loading older persisted states after packet contract changes

Success signal:
- state survives iteration without silent summary drift or alias churn

### 4. Verification And Proof Discipline

Current read:
- this axis is already close to Forge's current ceiling
- the main opportunity is simplification and consistency, not stricter gates

Why the score moved:
- browser proof and packet proof now sit closer to the same evidence story
- the release gate discipline from `1.12.0` stayed intact

`1.14.x` recommendation:
- formalize the proof taxonomy across red proof, rerun proof, browser proof, review proof, and residual risk
- clarify where each proof type lives and when it is considered fresh enough
- avoid adding more proof labels unless they change decision quality materially

Success signal:
- operators and adapters can tell which proof is missing without reconstructing the meaning of each field

### 5. Brownfield Safety And Scope Control

Current read:
- Forge is already strong here, but parallel packet work raises the bar
- the next risk is overlap and merge conflict management, not missing intent

Why the score moved:
- packet scope and ownership are more explicit
- blockers and residual risk now live closer to execution state

`1.14.x` recommendation:
- add stronger write-scope overlap checks before parallel or reviewer-lane dispatch
- standardize reopen rules when a packet's assumptions are invalidated by adjacent changes
- make merge-ready and review-ready criteria more conflict-aware

Success signal:
- packetized parallel work increases throughput without increasing brownfield surprise

### 6. Host-Neutral Dispatch Portability

Current read:
- `1.13.0` made real progress by separating dispatch mode from packet semantics
- the next step is to make host capability mapping easier to extend without contract drift

Why the score moved:
- `controller-sequential`, `independent-reviewer`, and `parallel-split` are now described through one packet contract
- adapter wording follows the contract more closely

`1.14.x` recommendation:
- formalize capability tiers, dispatch reasons, and fallback reasons in one registry-facing contract
- make it easier to add or tune a host without mutating core packet logic
- keep adapter wrappers explanatory, not semantic

Success signal:
- a new host can adopt Forge's dispatch model with minimal contract ambiguity

### 7. Browser QA And Runtime Proof Path

Current read:
- this axis improved the most in raw score because it started weakest
- Forge now treats browser QA as an execution decision, not a detached runtime trick

Why the score moved:
- packet eligibility classification exists
- runtime wrapper proof can flow back into workflow-state
- forwarding bugs in the runtime wrapper were fixed before release

`1.14.x` recommendation:
- add installed-runtime doctor or health verification for browser-capable toolchains
- improve retry and failure taxonomy so runtime failures do not look identical to missing proof
- make browser proof linking clearer across packet, runtime log, and summary surfaces

Success signal:
- browser-backed proof is dependable enough to accelerate delivery without becoming a trust tax

### 8. Operator Guidance Quality

Current read:
- `help` and `next` are now much closer to state-first operator tools than repo-first guessers
- remaining value is in sharper blocked-state and recovery guidance

Why the score moved:
- active packet, browser pending state, and next merge point now influence recommendations directly

`1.14.x` recommendation:
- add first-class blocked, stalled, and merge-conflict recovery guidance driven from workflow-state
- distinguish "resume current packet" from "prepare merge" from "rerun missing proof" even more sharply
- keep a single clear next action rather than widening into dashboards-in-text

Success signal:
- the operator gets the right next move even when the chain is stalled or partially merged

### 9. Release-Tail Maturity

Current read:
- this remains one of Forge's best-developed surfaces
- it should stay stable unless a build-side improvement genuinely changes release decisions

Why the score moved:
- only slightly, because `1.13.0` intentionally did not reopen the release-tail contract

`1.14.x` recommendation:
- preserve the current release-tier and readiness model
- only connect build packet outputs into release follow-up when they improve real release decisions
- reject any release-taxonomy growth that does not move the behavior bar

Success signal:
- release-tail stays clear and reliable while build-side gains leverage around it

### 10. Dashboard And Quick-Scan Visibility

Current read:
- the dashboard is finally useful because the underlying packet state is useful
- it is still correctly treated as a view, not an operating dependency

Why the score moved:
- active packets, browser QA pending, and next merge point now exist as meaningful fields to expose

`1.14.x` recommendation:
- only widen the dashboard when it reduces resume, review, or merge time measurably
- add filters or emphasis for blockers, merge points, and pending browser proof before adding any new summary fields
- continue to reject vanity telemetry, progress percentages, and pseudo-productivity metrics

Success signal:
- the dashboard shortens decision time without becoming the place where truth is invented

### 11. Docs And Reference Coherence

Current read:
- the doc set is coherent enough to support current behavior
- the next risk is manual drift as packet and runtime surfaces grow

Why the score moved:
- `target-state`, `reference-map`, `dashboard` reference, and public release wording are now more aligned with the build-side reality

`1.14.x` recommendation:
- generate more reference-facing structure from canonical registries or schemas where that reduces maintenance drift
- keep one canonical build-side explainer and avoid fragmenting behavior across workflow-only language
- treat docs as contract surfaces, not post-hoc narration

Success signal:
- reviewers can find contract truth quickly without reconciling multiple near-duplicates

## Weighted Headroom Ranking For 1.14.x

Weighted headroom uses current score and axis weight together to estimate where improvement still matters most:

```text
headroom index = axis weight * (10 - current score) / 10
```

Top `1.14.x` opportunities:

1. Browser QA and runtime proof path
2. Medium or large build execution
3. Small-task ergonomics
4. Host-neutral dispatch portability
5. Brownfield safety and scope control
6. Workflow-state backbone
7. Operator guidance quality
8. Release-tail maturity and dashboard visibility
9. Verification discipline
10. Docs and reference coherence

Interpretation:
- the next release should not chase the axes with the largest deltas from `1.12.0`
- it should chase the axes with the largest remaining weighted headroom after `1.13.0`

## Recommended 1.14.x Program

If Forge wants one coherent `1.14.x` theme, the strongest candidate is:

`execution throughput without contract drift`

That theme should bundle:

- stronger packet decomposition and merge protocol
- better host-capability portability
- runtime-tool and browser-proof reliability
- sharper fast-lane rules for truly small work
- more conflict-aware brownfield safeguards for packetized execution

It should not bundle:

- a new release-tier taxonomy
- dashboard vanity features
- broad companion sprawl as a substitute for core execution quality
- mandatory browser QA for non-browser work

## Guardrails For 1.14.x

- Keep `track_execution_progress.py` as the canonical packet source of truth unless there is a compelling migration plan.
- Do not let adapter wrappers become a second semantics layer.
- Do not solve runtime instability by weakening packet proof requirements.
- Do not let small-task optimization erase Forge's evidence-before-claims bar.
- If a new field matters operationally, put it in workflow-state first and expose it in dashboard or docs second.

## Recommended Exit Criteria For The Next Scorecard

Before calling `1.14.x` materially successful, the next scorecard should be able to claim:

- small-task ergonomics recovered without reducing proof quality
- packet decomposition and merge behavior are more explicit than in `1.13.0`
- browser-capable runtime paths are healthier and easier to trust
- host portability requires less adapter-specific explanation
- brownfield-safe packetized execution remains strong under more parallel or partially overlapping work
